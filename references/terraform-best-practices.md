# Terraform Best Practices

A curated summary of best practices for writing, organizing, and maintaining Terraform code.

## Key References

- [terraform-best-practices.com](https://www.terraform-best-practices.com/) by Anton Babenko — comprehensive guide covering structure, naming, and patterns
- [HashiCorp Terraform Style Guide](https://developer.hashicorp.com/terraform/language/style) — official style conventions
- [Terraform Module Registry](https://registry.terraform.io/) — reusable community modules
- [pre-commit-terraform](https://github.com/antonbabenko/pre-commit-terraform) — Git hooks for automated checks

## Code Organization

Structure Terraform projects by separating concerns into distinct modules and root configurations. Scale your layout based on project size:

- **Small projects**: single root module with all resources in one state file
- **Medium projects**: separate state files per environment; use published community modules
- **Large projects**: internal module registry, multiple accounts/regions, consider [Terragrunt](https://terragrunt.gruntwork.io/) for orchestration

Standard file layout for any module:

```
module/
├── main.tf        # resources and data sources (split into iam.tf etc. when >~150 lines)
├── variables.tf   # input variable declarations
├── outputs.tf     # output values
├── locals.tf      # local value expressions
├── data.tf        # data sources (merge into main.tf if trivial)
├── providers.tf   # provider + terraform blocks (root modules only)
├── versions.tf    # required_providers and version constraints
└── README.md      # generated by terraform-docs
```

For environment-specific values, use an `envs/` directory:

```
root/
├── main.tf
├── variables.tf
├── versions.tf
└── envs/
    ├── dev.tfvars
    ├── staging.tfvars
    └── prod.tfvars
```

**Never configure a provider inside a reusable module.** Modules should only declare `required_providers` in `versions.tf`. Provider configuration belongs exclusively in the root module's `providers.tf`.

For large multi-module repos, use a mono-repo per cloud provider with this layout:

```
repo/
├── modules/
│   ├── vpc/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   ├── outputs.tf
│   │   ├── versions.tf
│   │   └── README.md
│   └── compute/
├── examples/           # one subdirectory per reference architecture
│   ├── basic/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   ├── outputs.tf
│   │   ├── versions.tf
│   │   └── example.tfvars
│   └── advanced/
└── tests/
```

Pin module sources to a commit hash rather than a mutable semver tag for true immutability:

```hcl
module "lambda" {
  # Commit hash — cannot be overwritten, unlike a tag
  source = "github.com/terraform-aws-modules/terraform-aws-lambda.git?ref=e78cdf1f82944897ca6e30d6489f43cf24539374"
}
```

## State Management

- Store state remotely (S3 + DynamoDB, Azure Blob Storage, Terraform Cloud) from day one
- Enable state locking to prevent concurrent modifications
- Use one state file per environment (dev/staging/prod) and per concern (network, compute)
- Never commit `.tfstate` files to version control
- Mark sensitive outputs with `sensitive = true`

## Naming Conventions

- Use underscores (`_`) in Terraform identifiers; use hyphens in actual resource name strings
- Keep identifiers lowercase; use singular nouns for resource labels
- Do not repeat the resource type in its name (e.g., use `"public"` not `"public_route_table"` for `aws_route_table`)
- Follow the `{name}_{type}_{attribute}` pattern for output names (e.g., `private_subnet_id`)

## Variables and Outputs

- Always provide a `description` for every variable and output
- Use plural names for `list` and `map` types
- Use `nullable = false` for required variables; add `validation` blocks to enforce constraints
- Avoid double-negative names (prefer `encryption_enabled` over `encryption_disabled`)
- Add units to numeric variable names: `ram_size_gb`, `disk_size_mb`, `timeout_seconds`
- Don't provide defaults for environment-specific values (e.g., `account_id`); do provide defaults for environment-independent values (e.g., `disk_size_gb = 20`)
- Don't pass module outputs through input variables — reference resource attributes directly
- For outputs from a single resource, use a singular suffix (`virtual_network_id`). For outputs from `for_each`, use a plural suffix and a `for` expression to return a map:

```hcl
# Single resource
output "subnet_id" {
  description = "The ID of the created subnet."
  value       = aws_subnet.this.id
}

# Multiple resources (for_each)
output "subnet_ids" {
  description = "Map of subnet name to subnet ID."
  value       = { for k, v in aws_subnet.this : k => v.id }
}
```

## for_each vs count

Prefer `for_each` over `count` when creating multiple resources. `count` produces a list in state — removing an item from the middle forces Terraform to destroy and recreate everything after it. `for_each` produces a map keyed by a stable identifier, so individual resources can be added or removed safely.

```hcl
# BAD — count-based, index-sensitive
resource "aws_subnet" "this" {
  count      = length(var.subnets)
  cidr_block = var.subnets[count.index].cidr
  vpc_id     = aws_vpc.this.id
}

# GOOD — for_each over a map, stable keys
variable "subnets" {
  default = {
    "public"  = { cidr = "10.0.0.0/24", az = "us-east-1a" }
    "private" = { cidr = "10.0.1.0/24", az = "us-east-1b" }
  }
}

resource "aws_subnet" "this" {
  for_each          = var.subnets
  cidr_block        = each.value.cidr
  availability_zone = each.value.az
  vpc_id            = aws_vpc.this.id
}
```

Reserve `count` only for conditional single-resource creation:

```hcl
resource "aws_iam_role_policy_attachment" "this" {
  count      = var.enable_policy ? 1 : 0
  role       = aws_iam_role.this.name
  policy_arn = aws_iam_policy.this[0].arn
}
```

If your input is a list rather than a map, convert it first:

```hcl
locals {
  subnet_map = { for s in var.subnets : s.name => s }
}
```

For nested structures, use `flatten()` to produce a flat map suitable for `for_each`:

```hcl
# Input: map of route-table name -> list of routes
locals {
  routes = flatten([
    for rtb, rts in var.route_tables : [
      for rt in rts : merge(rt, { rtb = rtb })
    ]
  ])
}

resource "aws_route" "this" {
  for_each               = { for r in local.routes : "${r.rtb}-${r.destination_cidr}" => r }
  route_table_id         = aws_route_table.this[each.value.rtb].id
  destination_cidr_block = each.value.destination_cidr
}
```

## Variable Design Patterns

Design input variables to be intuitive and avoid forcing callers to keep parallel lists in sync. Prefer a map of objects as the top-level structure — the key enforces uniqueness and works directly with `for_each`:

```hcl
# BAD — disjoint parallel lists, order-sensitive
module "vpc" {
  subnet_names = ["public", "private"]
  subnet_cidrs = ["10.0.0.0/24", "10.0.1.0/24"]
}

# GOOD — map of objects, self-describing and for_each-ready
module "vpc" {
  subnets = {
    "public"  = { az = "us-east-1a", cidr = "10.0.0.0/24" }
    "private" = { az = "us-east-1b", cidr = "10.0.1.0/24" }
  }
}
```

Use `lookup(each.value, "optional_key", null)` for optional fields within the map to keep callers from having to specify every key.

For brownfield support (pre-existing resources), use `try()` to normalise between a managed `resource` and a `data` source:

```hcl
resource "azurerm_resource_group" "this" {
  count    = var.existing_rg ? 0 : 1
  name     = var.resource_group_name
  location = var.location
}

data "azurerm_resource_group" "this" {
  count = var.existing_rg ? 1 : 0
  name  = var.resource_group_name
}

locals {
  rg = try(azurerm_resource_group.this[0], data.azurerm_resource_group.this[0])
}
```

## Tagging

Apply consistent tags to every resource that supports them. Use the provider-level `default_tags` block to avoid repeating the same tags on every resource:

```hcl
provider "aws" {
  region = var.region

  default_tags {
    tags = {
      Environment = var.environment
      Project     = var.project
      ManagedBy   = "terraform"
      CostCenter  = var.cost_center
    }
  }
}
```

Resources can then add their own tags without repeating the defaults:

```hcl
resource "aws_instance" "web" {
  ami           = var.ami
  instance_type = var.instance_type

  tags = {
    Name    = "${var.project}-web"
    AppRole = "webserver"
  }
}
```

Recommended standard tags:

| Tag | Purpose |
|-----|---------|
| `Environment` | dev / staging / prod |
| `Project` | owning project or product |
| `ManagedBy` | terraform |
| `CostCenter` | billing allocation |
| `AppRole` | technical function (webserver, database) |

## Security & IAM Auth

**Use IAM roles, not static access keys.** Configure the AWS provider to assume a role — credentials are short-lived and automatically rotated:

```hcl
provider "aws" {
  region = var.region

  assume_role {
    role_arn     = "arn:aws:iam::111122223333:role/terraform-execution"
    session_name = "terraform-${var.environment}"
  }
}
```

Trust relationship for the role (allows another role to assume it):

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "AWS": "arn:aws:iam::ACCOUNT:role/ci-runner" },
    "Action": "sts:AssumeRole"
  }]
}
```

**For CI/CD pipelines, use OIDC federation** instead of long-lived keys:

```hcl
# GitHub Actions — exchange GitHub OIDC token for temporary AWS credentials
resource "aws_iam_openid_connect_provider" "github" {
  url             = "https://token.actions.githubusercontent.com"
  client_id_list  = ["sts.amazonaws.com"]
  thumbprint_list = ["6938fd4d98bab03faadb97b34396831e3780aea1"]
}
```

**Use Sentinel policies** to enforce organizational guardrails (HCP Terraform / Terraform Enterprise):

- Require specific tags on all resources
- Restrict instance types to an approved list
- Prevent destruction of production resources
- Enforce mandatory variable values

**Use AWS Secrets Manager** for sensitive values — never store secrets in plaintext in state. Mark sensitive outputs to suppress CLI display:

```hcl
output "db_password" {
  value     = aws_db_instance.this.password
  sensitive = true
}
```

## Module Composition

**Never pass an entire module call as an object.** Referencing `module.x` as a value (through a local, variable, or output) causes unexpected cascading deletes: when any resource inside the module is destroyed, all resources that depend on the module object are also destroyed, even without a real dependency.

```hcl
# BAD
locals {
  config = module.networking          # propagates deletes unexpectedly
}

# GOOD — reference specific outputs
locals {
  vpc_id     = module.networking.vpc_id
  subnet_ids = module.networking.subnet_ids
}
```

**Use attachment resources instead of inline blocks.** Inline blocks (e.g., `ingress`/`egress` inside `aws_security_group`) conflict with separately managed rules and cause unexpected diffs. Use standalone attachment resources instead:

```hcl
# BAD — inline ingress block conflicts with aws_security_group_rule
resource "aws_security_group" "web" {
  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# GOOD — separate attachment resource
resource "aws_security_group" "web" {
  name = "${var.project}-web"
}

resource "aws_security_group_rule" "https_in" {
  type              = "ingress"
  from_port         = 443
  to_port           = 443
  protocol          = "tcp"
  cidr_blocks       = ["0.0.0.0/0"]
  security_group_id = aws_security_group.web.id
}
```

**Segment resources to avoid forced recreation.** Some cloud providers destroy and recreate a parent resource when a child attribute changes. Extract the child into a separate resource block:

```hcl
# Separate network interface from instance so attaching/detaching
# an interface does not destroy the EC2 instance.
resource "aws_instance" "this" {
  for_each = var.instances
  ami           = var.ami
  instance_type = var.instance_type

  network_interface {
    device_index         = 0
    network_interface_id = aws_network_interface.primary[each.key].id
  }
}

resource "aws_network_interface" "primary" {
  for_each  = var.instances
  subnet_id = each.value.subnet_id
}
```

## Pre-Commit Hooks as a CI Gate

[pre-commit-terraform](https://github.com/antonbabenko/pre-commit-terraform) bundles tflint, terraform-docs, Checkov, Trivy, and Terrascan into local git hooks. Run these same hooks in CI to prevent any bypass:

```yaml
- name: Run pre-commit
  run: pre-commit run --all-files
```

Key tools to install before running pre-commit in CI: `trivy`, `terraform-docs`, `tflint`, `terrascan` (each must be on `PATH`). Use pinned versions via env vars to keep CI and local environments consistent:

```yaml
env:
  TERRAFORM_DOCS_VERSION: v0.19.0
  TFLINT_VERSION: v0.55.1
```

## Testing

Use Go-based [Terratest](https://terratest.gruntwork.io/) for integration tests. Place all test cases under `tests/` at the repo root, one subdirectory per case. Use these suffixes to communicate intent:

| Suffix | Meaning |
|--------|---------|
| `_read` | Reads pre-existing (brownfield) resources |
| `_mod`  | Applies config, then applies a modified version |
| `_e2e`  | Applies resources and exercises them end-to-end |
| `_plan` | Plan-only; never calls Apply (unit tests) |

Run with `-count 1` to bypass Go's test cache, which is incompatible with Terraform:

```bash
go test -count 1 ./tests/...
```

### Unit tests (plan-only)

The most common Terraform unit test validates that the module handles **unknown values** correctly. Terraform marks resource attributes as "unknown" during Plan until Apply resolves them. Code that works with known values can silently fail with unknown ones.

Use `random_pet` to cheaply generate unknown values without touching any cloud API:

```hcl
resource "random_pet" "this" {}

locals {
  u_string = random_pet.this.id
  u_number = length(random_pet.this.id)
  u_bool   = length(random_pet.this.id) > 0   # always true, but unknown at plan time
}
```

Pass these locals as module inputs to verify the module plans cleanly with unknown values.

For map outputs, verify they can be used as `for_each` keys (unknown keys fail `for_each`):

```hcl
# If this plans successfully, the output keys are known — safe for downstream for_each
resource "random_pet" "consume" {
  for_each = merge(
    module.mymodule.subnet_ids,
    module.mymodule.route_table_ids,
  )
}
```

## Workspace Organization

In HCP Terraform or Terraform Enterprise, create **one workspace per infrastructure component per environment** — not one workspace per environment.

```
# Pattern: <component>-<environment>
networking-dev
networking-staging
networking-prod
billing-app-dev
billing-app-staging
billing-app-prod
```

Benefits:
- Blast radius is scoped to a single component
- Teams receive access only to the components they own
- Parallel changes across components without state contention
- Clear ownership — misaligned workspace structure usually means unclear ownership

Access control maps to organizational roles:

| Persona | Access |
|---------|--------|
| Central IT | Read-only across all workspaces; policy enforcement |
| Organization Architect | Status overview, dependency graphs |
| Workspace Owner | Full lifecycle across all environments for their component |
| Workspace Contributor | Apply to non-production; propose changes to production |
